# Feature: Deployment (Render + Cloudflare + Resend)

- **ADR:** [0049](../adr/0049-hosting-render-cloudflare-resend.md) · **Status:** dev environment
- **Stack after ADR 0048:** web (Astro SSR) + API (NestJS) + managed Postgres — no Meili/MinIO.

## Render Blueprint (`render.yaml`)

Three resources, deployed from GitHub `TheY2T/the-music-repository` into the **`dev` env group**:

| Resource | Notes |
|---|---|
| `tmr-db-dev` | Managed Postgres 16 |
| `tmr-api-dev` | Docker (`apps/api/Dockerfile`), port 3000, health `/health` |
| `tmr-web-dev` | Docker (`apps/web/Dockerfile`), port 4321 |

- **Migrations/seed** run in the API's `preDeployCommand`: `db:migrate && db:seed:auth && (db:seed || true)`.
  The runner image contains `drizzle-kit` + the `drizzle/` folder (it copies the full installer output).
- **Web ↔ API (server-side)** uses Render private networking: `API_INTERNAL_URL=http://tmr-api-dev:3000`.
  This is not affected by the Cloudflare Access gate on the public hostnames.
- **Build-time public config** is baked into the web image via Dockerfile `ARG`s (Render passes matching
  service env vars as build args): `PUBLIC_SITE_URL`, `PUBLIC_API_BASE_URL`.
- **Secrets** (`SMTP_URL`, `MAIL_FROM`, …) live in the shared `dev` env group (`fromGroup: dev`);
  `BETTER_AUTH_SECRET` is generated by Render.

## Cloudflare

- **DNS:** `themusicrepository.com` → web service, `api.themusicrepository.com` → API service (Render
  custom-domain CNAMEs); SSL Full (strict).
- **Cloudflare Access (Zero Trust):** one Access application over both hostnames, allowing only the
  operator's email — keeps the dev environment private with no app code.
- **De-indexing (defence in depth):** with `APP_ENV=dev`, `BaseLayout` emits a global `noindex,nofollow`
  and `robots.txt` returns `Disallow: /`. Flipping `APP_ENV=production` restores indexing.
- **CDN caching:** the origins compress responses and set `Cache-Control`; Cloudflare Cache Rules (Tiered
  Cache, auth-cookie bypass) activate edge caching. See [caching.md](./caching.md) + ADR 0051; the rules
  are applied per the deploy runbook.

## Resend + contact form

- Resend sends via the existing SMTP path: `SMTP_URL=smtps://resend:<RESEND_API_KEY>@smtp.resend.com:465`,
  `MAIL_FROM="The Music Repository <admin@themusicrepository.com>"` — set in the `dev` group. No new
  mail code (Better Auth account emails also flow through it).
- Add the SPF/DKIM/DMARC + return-path records Resend generates into Cloudflare DNS and verify the domain.
- Contact form: see `contact.md`. Protect it with **Cloudflare Turnstile** — create a widget in the
  Cloudflare dashboard, then set `TURNSTILE_SECRET_KEY` (API) + `PUBLIC_TURNSTILE_SITE_KEY` (web build)
  in the `dev` env group. Unset ⇒ the challenge is skipped.

## Cross-subdomain auth

`AUTH_COOKIE_DOMAIN=.themusicrepository.com` shares the Better Auth session cookie across the apex and
`api.` (`SameSite=None; Secure`). `TRUSTED_ORIGINS`/`BETTER_AUTH_URL` name the real HTTPS origins. Unset
locally, where host-only lax cookies over http work.

## Manual steps (operator)

1. Cloudflare: confirm the domain's nameservers are on Cloudflare; create the Access application + policy.
2. Resend: create the account + API key; add the DNS records; verify the domain and `contact@` sender.
3. Render: connect the repo / apply the Blueprint; set `SMTP_URL` + `MAIL_FROM` in the `dev` group; add
   the custom domains and copy the CNAME targets into Cloudflare DNS.

## Verify

- Blueprint deploys web + API + Postgres; `preDeployCommand` migrations succeed.
- Cloudflare Access challenges both hostnames; `/robots.txt` is `Disallow: /`; page source carries the
  global `noindex`. Sign in on the apex → the session cookie reaches `api.` (cross-subdomain auth).
- Submit the contact form → email arrives at the operator from `admin@themusicrepository.com`.
