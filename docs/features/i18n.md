# Feature: Localization / i18n (English + 中文 zh-Hans)

- **Phase:** i18n · **Status:** shipped (web UI strings)
- **Flag key:** `platform.i18n` (from `@TheY2T/tmr-flags`)
- **ADR:** 0017

## Purpose

Make the whole web UI **locale-aware** so non-English learners can use the app. First two languages:
**English** (default) and **Simplified Chinese** (`zh-Hans`), extensible to more later. Scope of this
pass is **web UI strings only** — backend problem+json/mail strings and DB catalogue content are
deferred (see §Scope).

## Packages (single sources of truth — never duplicate)

- **`@TheY2T/tmr-i18n-locales`** (`packages/i18n-locales/`) — the raw JSON catalogues, one file per
  language (`src/en.json`, `src/zh-Hans.json`). **`en.json` is the source of truth for message keys**:
  `type MessageKey = keyof typeof en`, so a new key is added to `en.json` first and TypeScript then
  flags any consumer using an unknown key. Other languages are `Partial` (missing key → English at
  runtime). tsup dual ESM+CJS (like `@TheY2T/tmr-flags`).
- **`@TheY2T/tmr-i18n`** (`packages/i18n/`) — the dependency-free engine: `LOCALES`, `Locale`,
  `DEFAULT_LOCALE`, `t(locale, key, params?)` (with `{name}` interpolation + English fallback),
  `localizedPath(locale, path)`, `splitLocalePath`, `htmlLang`, `localePrefix`, `preferredLocale`,
  `isLocale`, `LOCALE_LABELS`. Depends on `-i18n-locales`.

Both are turbo `^build` deps of `apps/web` (they build before web typechecks/builds).

## Routing (URL-prefix; single page-file set)

- English at `/`, Chinese at `/zh/…`. **One set of page files serves every locale** — the locale is not
  a route directory.
- `apps/web/src/middleware.ts` resolves `Astro.locals.locale` per request. For a `/zh/…` request it
  sets `locale = zh-Hans` and **rewrites** to the canonical (un-prefixed) path via `next(canonicalPath)`
  so the single page file renders while the browser URL stays `/zh/…`. On the bare root `/` only, a
  saved `locale` cookie / `Accept-Language` redirects to the prefixed URL.
- Astro's built-in `i18n` config is **not** used (it would fight the single-file rewrite). `site` is set
  in `astro.config.mjs` for absolute `hreflang` URLs.
- Gated by `platform.i18n`: off ⇒ everything serves English, the switcher is hidden, `/zh/*` is not
  routed.

## Rendering

- **Shared shell:** `apps/web/src/layouts/BaseLayout.astro` — dynamic `<html lang>` (+ a `lang-zh` class
  for the CJK font stack in `global.css`), the pre-paint dark-mode script, `hreflang` alternates
  (`en` / `zh-Hans` / `x-default`), and a top bar hosting the **`LanguageSwitcher.tsx`** island (sets a
  1-year `locale` cookie, links to the same page in the other locale). Every page renders its `<main>`
  into the default slot.
- **`.astro` pages:** `const locale = Astro.locals.locale;` then `{t(locale, 'key')}`. Build internal
  links with `localizedPath(locale, '/path')` so a zh page stays in Chinese. Build the `<title>` inline
  (e.g. `` `${t(locale, 'nav.drills')} — ${t(locale, 'site.name')}` ``).
- **React islands:** receive `locale: Locale` as a **plain prop** (the only new prop), `import { t }
  from '@TheY2T/tmr-i18n'`, and call `t(locale, 'key')`. **No React context** — it can't cross island
  boundaries (see `apps/web/CLAUDE.md`), and the locale is deterministic from the URL so the island
  hydrates with the same string the server rendered (no flash / no hydration mismatch). Catalogues are
  static-imported into the island bundle (a shared Vite chunk), so only the short `locale` string is
  serialized per island.

## Key naming

`domain.thing` in camelCase after the dot (e.g. `nav.catalogue`, `catalogue.search`, `upgrade.subtitle`).
Tools share `tool.<slug>.title` / `tool.<slug>.summary` between the `/tools` hub and each tool page.
Island keys use a per-island prefix (`prog.*`, `classmgr.*`, `cform.*`, …).

## What is translated vs left as-is

- **Translated:** all UI chrome — nav, buttons, labels, placeholders, `<title>`s, headings,
  descriptions, status/empty/error copy, form field labels.
- **Left language-neutral (by design):** music-theory tokens (note names C/D/E, chord symbols, Roman
  numerals, degrees), technical names (MusicXML, MEI, Verovio, MIDI, A440, CAGED, drop-2/3), and
  **API/DB data** (catalogue titles/summaries/bodies, taxonomy names, classroom/user names) — the
  latter is a separate backend/DB concern (deferred).

## Adding a language

1. Add `src/<locale>.json` to `@TheY2T/tmr-i18n-locales` and export it from `src/index.ts`.
2. Register the locale in `@TheY2T/tmr-i18n` (`LOCALES`, `URL_PREFIX`, `LOCALE_LABELS`, `htmlLang` if the
   `<html lang>` differs from the id) and add its `Accept-Language` match in `matchAcceptLanguage`.
3. Rebuild the packages. The switcher and routing pick it up automatically.

See the `add-translations` skill for adding/using individual strings.

  Chinese chrome (catalogue **data** stays in its stored language — expected); islands hydrate with no

## Scope & next phases

- **This pass:** web UI strings (en + zh-Hans), URL-prefix routing, shared layout + switcher.
- **Incremental (small, left English for now):** deep admin authoring-form option *values*, client-side
  drill **deck names/descriptions** (`src/lib/drill-decks.ts`), and imperative admin redirects that drop
  the `/zh` prefix.
- **Next (deferred):** backend string i18n — `Accept-Language` on the ALS `RequestContext`, a
  `code`-keyed localized error catalog in `ProblemDetailsExceptionFilter`, localized mail templates.
- **Later (largest):** multilingual **DB content** — per-locale translation tables for
  `content_items`/`collections`/`help_topics`/taxonomy + CMS + Meilisearch per-locale reindex + spec
  changes; api-client passes the negotiated locale to the API.
