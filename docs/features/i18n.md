# Feature: Localization / i18n (English + 中文 zh-Hans)

- **Phase:** i18n · **Status:** shipped (web UI strings; DB-backed + admin-managed)
- **Flag keys:** `platform.i18n` (routing + switcher), `admin.locale-strings` (the CMS surface)
- **ADR:** 0017 (routing) + **0034 (DB-backed strings via the admin CMS)**

## Purpose

Make the whole web UI **locale-aware** so non-English learners can use the app (English default +
Simplified Chinese `zh-Hans`, extensible), and let admins **edit UI copy and translations at runtime**.
UI strings live in the **database** and are managed from the admin CMS; the in-repo JSON is
the **seed baseline + compile-time key type + last-resort fallback**.

## Storage: database is the runtime source of truth (ADR 0034)

- **`ui_messages`** — one row per `(locale, key)` with a `draft_value` / `published_value` pair, `status`
  (draft|published), `seeded` (true = baseline, false = runtime-edited), and `deleted_at` (soft delete).
  Only **published, non-deleted** rows are served. `ui_message_revisions` logs every change (diff +
  restore); `i18n_versions` holds the per-locale version tag (epoch-ms of last publish) used as the ETag
  + cache-bust signal.
- **Seed:** `apps/api/src/infrastructure/database/seed-i18n.ts` upserts `en.json` + `zh-Hans.json` as
  published `seeded` rows on `db:seed` — a fresh deploy always boots with a full catalogue. Idempotent
  (never clobbers admin edits; only fills missing baseline keys).
- **Admin CMS:** the **Localization** section at `/admin/localization` (gated by `admin.locale-strings` +
  `guardAdmin`, island `packages/common-ui/src/AdminLocaleManager.tsx`) manages the **general-site UI
  strings** — a table **grouped by key**; editing a key opens a modal that manages every locale's
  translation (add / edit / remove / restore); soft delete + restore; **Publish** promotes drafts +
  cache-busts. Search (client-side, deferred) + status/deleted filters + pagination. Client wrappers:
  `packages/web-data/src/i18n-api.ts`. **Content** localization (catalogue / collection / help item text)
  is authored **inline in each entity's own editor** — see §Phase 2.
- **Locale registry (`locales` table):** the CMS can author strings for **any locale code** — admins
  create new locales in the UI, and **import/export** a locale's strings as flat key→value JSON (export is
  filterable by origin: baseline vs runtime). This set is a **superset** of the code-level routing locales
  (`LOCALES` in `@TheY2T/tmr-i18n`): a new locale is translatable immediately, but wiring it into URL
  routing (`/fr`) + the language switcher + the `Locale` type is still a **deploy** (see §Adding a language).

## Packages (the engine + the seed/type/fallback source)

- **`@TheY2T/tmr-i18n-locales`** (`packages/i18n-locales/`) — the raw JSON catalogues (`src/en.json`,
  `src/zh-Hans.json`). The **DB seed baseline**, the
  **compile-time key type** (`type MessageKey = keyof typeof en`), and the **bundled fallback**. tsup dual
  ESM+CJS.
- **`@TheY2T/tmr-i18n`** (`packages/i18n/`) — the engine. `t(locale, key, params?)` resolves against a
  **mutable per-locale registry** (`loadCatalogue` / `getCatalogueVersion` / `hydrateCatalogueFromDom`),
  falling back to the bundled catalogue then the key (never blank). Also exports `LOCALES`, `Locale`,
  `DEFAULT_LOCALE`, `localizedPath`, `splitLocalePath`, `htmlLang`, `localePrefix`, `preferredLocale`,
  `isLocale`, `LOCALE_LABELS`, and the `SerializedCatalogue` blob type.

Both are turbo `^build` deps of `apps/web`. `apps/api` also depends on `-i18n-locales` (for the seed).

## Routing (URL-prefix; single page-file set)

- English at `/`, Chinese at `/zh/…`. **One set of page files serves every locale** — the locale is not
  a route directory.
- `apps/web/src/middleware.ts` resolves `Astro.locals.locale` per request. For a `/zh/…` request it
  sets `locale = zh-Hans` and **rewrites** to the canonical (un-prefixed) path via `next(canonicalPath)`
  so the single page file renders while the browser URL stays `/zh/…`. On the bare root `/` only, a
  saved `locale` cookie / `Accept-Language` redirects to the prefixed URL.
- Astro's built-in `i18n` config is **not** used (it would fight the single-file rewrite). `site` is set
  in `astro.config.mjs` for absolute `hreflang` URLs.
- Gated by `platform.i18n`: off ⇒ everything serves English, the switcher is hidden, `/zh/*` is not
  routed.

## Rendering

- **Shared shell:** `apps/web/src/layouts/BaseLayout.astro` — dynamic `<html lang>` (+ a `lang-zh` class
  for the CJK font stack in `global.css`), the pre-paint dark-mode script, `hreflang` alternates
  (`en` / `zh-Hans` / `x-default`), and a top bar hosting the **`LanguageSwitcher.tsx`** island (sets a
  1-year `locale` cookie, links to the same page in the other locale). Every page renders its `<main>`
  into the default slot.
- **`.astro` pages:** `const locale = Astro.locals.locale;` then `{t(locale, 'key')}`. Build internal
  links with `localizedPath(locale, '/path')` so a zh page stays in Chinese. Build the `<title>` inline
  (e.g. `` `${t(locale, 'nav.drills')} — ${t(locale, 'site.name')}` ``).
- **React islands:** receive `locale: Locale` as a **plain prop**, `import { t } from '@TheY2T/tmr-i18n'`,
  and call `t(locale, 'key')`. **No React context** — it can't cross island boundaries (see
  `apps/web/CLAUDE.md`), and the locale is deterministic from the URL so the island hydrates with the same
  string the server rendered (no flash / no hydration mismatch).

## Catalogue delivery (how DB strings reach `t()`) — ADR 0034

`t()` reads a mutable module-level registry, populated in each of the two worlds so all ~125 call sites
stay unchanged:

- **Server (SSR):** `apps/web/src/lib/i18n-catalogue.ts` caches each locale's published catalogue in
  web-process memory, revalidating cheaply against the API's `GET /i18n/version` (short TTL) and only
  re-fetching `GET /i18n/catalogue/{locale}` when the version changed. `apps/web/src/middleware.ts` calls
  `ensureCatalogue(locale)` before render (which calls `loadCatalogue`, so SSR `t()` resolves from the DB)
  and sets `Astro.locals.i18nCatalogue`.
- **Client:** `BaseLayout.astro` serializes that catalogue into a `<script id="i18n-catalogue"
  type="application/json">` blob, and a `<head>` module calls `hydrateCatalogueFromDom()` **before any
  island hydrates** — so browser `t()` resolves from the same strings. If the blob/API is missing, the
  engine falls back to the bundled catalogue, so the page always renders.
- **Cache-bust:** publishing bumps the locale's `i18n_versions` row; the API cache + web cache converge
  within seconds, so edits go live.

## Key naming

`domain.thing` in camelCase after the dot (e.g. `nav.catalogue`, `catalogue.search`, `upgrade.subtitle`).
Tools share `tool.<slug>.title` / `tool.<slug>.summary` between the `/tools` hub and each tool page.
Island keys use a per-island prefix (`prog.*`, `classmgr.*`, `cform.*`, …).

## What is translated vs left as-is

- **Translated:** all UI chrome — nav, buttons, labels, placeholders, `<title>`s, headings,
  descriptions, status/empty/error copy, form field labels.
- **Left language-neutral (by design):** music-theory tokens (note names C/D/E, chord symbols, Roman
  numerals, degrees), technical names (MusicXML, MEI, Verovio, MIDI, A440, CAGED, drop-2/3), and
  **API/DB data** (catalogue titles/summaries/bodies, taxonomy names, classroom/user names) — the
  latter is a separate backend/DB concern (deferred).

## Editing / adding strings

- **Change wording or a translation for an existing key, or add a runtime-only string:** do it in the
  **admin CMS** (`/admin/locale-strings`) → edit/create → **Publish**.
- **Add a brand-new key that typed code references:** add it to `en.json` (+ `zh-Hans.json`) so
  `MessageKey` widens and the key seeds; ship once; thereafter its wording is CMS-editable. See the
  `add-translations` skill.

## Adding a language

1. Add `src/<locale>.json` to `@TheY2T/tmr-i18n-locales` (seed baseline) and export it from `src/index.ts`.
2. Register the locale in `@TheY2T/tmr-i18n` (`LOCALES`, `URL_PREFIX`, `LOCALE_LABELS`, `htmlLang` if the
   `<html lang>` differs from the id) and add its `Accept-Language` match in `matchAcceptLanguage`; add it
   to `CATALOGUES` in `seed-i18n.ts` and the `Locale` union in `packages/web-data/src/types.ts`.
3. Rebuild the packages and re-seed (`db:seed`). Thereafter its strings are CMS-managed like any other.

See the `add-translations` skill for adding/using individual strings.

## Verification

- `pnpm build lint check-types test` green; `zh-Hans` has no missing keys vs `en` in the **seed**
  (the parity test in `packages/i18n-locales` + the `MessageKey` type surface gaps at build).
- API: `curl localhost:3000/i18n/version` and `.../i18n/catalogue/en` serve the seeded catalogue (ETag
  present; `If-None-Match` → 304).
- Browser: an admin edits a string at `/admin/locale-strings` → **Publish** → the change appears on a
  public page within seconds; type-the-key delete + restore work; `/` vs `/zh/` render the
  right locale; islands hydrate with no flash; stopping the API still renders (bundled fallback).

## Scope & next phases

- **Shipped:** web UI strings (en + zh-Hans) — **DB-backed + admin-managed** (ADR 0034), URL-prefix
  routing, shared layout + switcher.
- **Incremental (small, left English for now):** deep admin authoring-form option *values*, client-side
  drill **deck names/descriptions** (`src/lib/drill-decks.ts`), and imperative admin redirects that drop
  the `/zh` prefix.
- **Next (deferred):** backend string i18n — `Accept-Language` on the ALS `RequestContext`, a
  `code`-keyed localized error catalog in `ProblemDetailsExceptionFilter`, localized mail templates.
- **Phase 2 — multilingual DB content (`entity_translations`):**
  - **Read overlay:** per-locale content overlay at read time via `?locale=`, for **catalogue** (title /
    summary / bodyMdx / `details.<field>` on detail + related), **collections** (title / summary /
    bodyMdx / curatorBio + `section.<id>.title|description` + `outcome.<i>` + `item.<slug>.curatorNote`,
    plus item summaries) and **help topics** (term / body). Backend `apps/api/src/translations/`
    (`ContentTranslations` overlay port + `EntityTranslationAuthoring` write side, draft→publish→revisions
    + soft delete, publish scopes to an optional `locale`); the read use-cases apply pure overlay helpers
    (`applyContentOverlay`/`applyCollectionOverlay`); the web threads `Astro.locals.locale` into the
    content/collection hooks (help `InfoView` takes an optional `locale`).
  - **Authoring UI (ADR 0036):** localization is **integrated into each entity editor** — a **locale
    switcher** (`LocaleBar`) in the editor header rebinds the *same* properties + body form to the active
    locale. The base locale (`en`) edits the entity; a target locale edits its translation with the
    English original shown for reference and structural fields (slug/type/taxonomy…) locked. Scope covers
    title/summary/rich-body plus **metadata**: the content `details` facts (editable base inputs + per-locale
    translation) and collection **sections / outcomes / per-item notes**. Each locale has its own
    draft→publish (`Save`/`Publish translation`), independent of the base entity, and a second locale can be
    authored at create time (buffered, flushed after the base is created). Gated by the `localeStrings` flag.
    Shared pieces: `packages/common-ui/src/localization/` (`useLocaleContent`, `LocaleBar`,
    `LocalizableField`); field registry + base-value resolution in
    `packages/web-data/src/content-translations-api.ts` (`LOCALIZABLE_FIELDS`, `getTranslationTarget`).
    Detail endpoints expose `id` so the editor can key `entity_translations`.
  - **Remaining:** taxonomy names, **Meilisearch per-locale search** (search results still come from the
    base index; translation-publish deliberately doesn't reindex yet), the catalogue **list** overlay, and
    wiring `InfoView`'s `locale` at its tool-page mounts.
