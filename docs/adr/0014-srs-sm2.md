# ADR 0014 — Spaced repetition via SM-2, server-scheduled

- **Status:** Accepted
- **Context:** Phase 4 turns the interactive tools into trainers. Drills (interval/chord recognition,
  etc.) need durable, per-user scheduling so learners review the right cards at the right time. We need
  an algorithm, a place to run it, and a card model that doesn't hard-code every drill in the database.
- **Decision:**
  - **Algorithm: SM-2** (the classic SuperMemo-2). Pure, well-understood, no dependencies. Implemented
    framework-free in `apps/api/src/reviews/domain/sm2.ts` (`applySm2(state, quality, now)`): grades <3
    lapse the card (reset reps, due tomorrow); grades ≥3 advance the interval (1 → 6 → prev×ease) and
    adjust the ease factor (floor 1.3). FSRS can replace it later behind the same port.
  - **Server-scheduled, client-defined decks.** The backend stores only **scheduling state** keyed by
    `(userId, deck, card)` — it never needs to know what a card *is*. **Decks are defined client-side**
    (`apps/web/src/lib/drill-decks.ts`): the card key + how to render/play the question + the answer
    label. This keeps question generation next to the audio/theory code (Phase 3) and lets new decks
    ship without a migration.
  - **API (spec-first, `@RequireAuth`):** `GET /me/reviews` (per-deck learned/due counts),
    `GET /me/reviews/{deck}` (stored states — the client computes new vs due from its deck definition),
    `POST /me/reviews/{deck}/{card}` `{quality}` → applies SM-2, returns the updated state. User id comes
    from the `CurrentUser` port.
  - **Data:** `review_cards` (`ease_factor` double, `interval_days`, `repetitions`, `due_at`,
    `last_reviewed_at`; PK `(user_id, deck, card)`). Migration `drizzle/0007_*`.
  - **Session model:** the client builds a session queue of **due** (`due_at ≤ now`) + **new** (not yet
    stored) cards, capped per session; each answer POSTs a grade.
- **Consequences:** durable multi-deck SRS with a tiny, generic backend; drills are pure client data.
  The algorithm is one pure function and one port, so swapping SM-2 → FSRS, or adding decks
  (note-name, key-signature, scale-degree), is cheap. Reference: SuperMemo SM-2.
