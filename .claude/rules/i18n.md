# Localization / i18n (ADR 0017 + 0034, `docs/features/i18n.md`)

Web UI is English + 中文 (`zh-Hans`), URL-prefix routed (`/zh/…`), gated by `platform.i18n`. String
**values are DB-backed and edited in the admin CMS** under the **Localization** section
(`/admin/localization`, flag `admin.locale-strings`) with no redeploy (ADR 0034); the in-repo JSON is the
DB **seed** + compile-time key type + fallback. Follow the **`add-translations`** skill.

- **No hardcoded UI strings.** Every user-facing string goes through `t(locale, key)` from
  `@TheY2T/tmr-i18n`. To change wording/translations, edit in the **CMS** and publish. A brand-new
  key referenced by typed code must be added to `@TheY2T/tmr-i18n-locales` (**`en.json` is the key type
  source** → `type MessageKey`, and the seed; `zh-Hans.json` the Chinese seed) so it types + seeds; ship
  once, then it is CMS-editable.
- **Runtime delivery:** `t()` reads a mutable registry loaded from the DB — server-side in
  `middleware.ts` (`ensureCatalogue`) and client-side in `BaseLayout.astro`
  (`hydrateCatalogueFromDom`). Don't reintroduce static catalogue imports in the render path.
- **Locale** resolves per request in `middleware.ts` → `Astro.locals.locale` (URL prefix > cookie >
  Accept-Language). `/zh/…` rewrites to the canonical page path (single page-file set).
- **Islands** take `locale: Locale` as a **plain prop** (deterministic from the URL → no hydration flash;
  React context can't cross islands) and call `t(locale, key)`. Build internal links with
  `localizedPath(locale, '/path')`; build `<title>` inline.
- **No glyphs in strings** — icons come from the `Icon` atom, not baked into i18n text (see
  `.claude/rules/design-system.md`).
- **Content localization (Phase 2, `entity_translations`):** **catalogue, collections, and help** content
  is overlaid per-locale at read time via `?locale=` — authored in the CMS, served through the
  `ContentTranslations` port injected into each read path (`apps/api/src/translations/`), applied by pure
  `apply*Overlay` domain helpers. **UI:** content localization lives **inside each entity's own editor** —
  the `ContentLocalizationEditor` panel is embedded as a collapsible **Localization** section in
  `ContentForm` / `CollectionForm` / `HelpTopicForm` (edit mode, gated by the `localeStrings` flag), with a
  locale tab strip, the block editor for body fields, and the English original for reference; it saves +
  publishes translations independently of the base entity. The standalone **Localization** admin area
  (`/admin/localization`, `AdminLocaleManager`) is now **only** the general-site UI strings. Still
  base-only: taxonomy names, the catalogue **list**, and Meilisearch search results (per-locale index TBD;
  translation-publish doesn't reindex).
- **Left language-neutral:** music-theory tokens + technical names.
